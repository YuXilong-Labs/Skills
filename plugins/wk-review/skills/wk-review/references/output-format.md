# Review 报告输出格式

## 报告模板

```markdown
# 📋 Code Review 报告

## 修改概要

| 项目 | 值 |
|------|------|
| **审查范围** | `{scope}` |
| **变更文件数** | {count} |
| **新增行数** | +{additions} |
| **删除行数** | -{deletions} |
| **审查时间** | {date} |

### 变更文件列表

| 文件 | 变更类型 | +/- |
|------|----------|-----|
| `path/to/file1.swift` | 修改 | +15 / -3 |
| `path/to/file2.swift` | 新增 | +120 / -0 |

---

## 问题列表

### 🔴 CRITICAL（{count}）

#### 1. {问题标题}

**文件：** `{file_path}:{line_number}`
**维度：** {审查维度，如：crash / logic / memory}

**问题描述：**
{简洁说明这是什么问题}

**代码证据：**
```{language}
// {file_path}:{start_line}-{end_line}
{引用的原始代码片段}
```

**分析依据：**
{解释为什么这是一个问题，引用具体逻辑或规则}

**修复建议：**
```{language}
{修复后的代码示例}
```

---

### 🟠 HIGH（{count}）

#### 1. {问题标题}
{同上格式}

---

### 🟡 MEDIUM（{count}）

#### 1. {问题标题}
{同上格式}

---

### 🟢 LOW（{count}）

#### 1. {问题标题}
{同上格式}

---

## 影响分析

### 行为变更
{列出本次修改可能导致的行为变化}

### 需关注的依赖
{列出被修改代码的外部调用者或依赖方}

---

## 总结

| 级别 | 数量 |
|------|------|
| 🔴 CRITICAL | {count} |
| 🟠 HIGH | {count} |
| 🟡 MEDIUM | {count} |
| 🟢 LOW | {count} |
| **合计** | **{total}** |

### 建议
{给出总体建议：是否可以安全合并、哪些问题需要优先修复}
```

---

## 格式规则

1. **每个问题都必须包含完整的 5 个部分**：文件行号、维度、问题描述、代码证据、修复建议
2. **代码证据必须是原始代码**：从文件中复制，不要重写或简化
3. **行号必须精确**：使用 `file:line` 或 `file:start-end` 格式
4. **修复建议尽可能给出代码**：而不是仅文字描述
5. **问题为空的级别可以省略**：如无 CRITICAL 问题，跳过该段
6. **影响分析可选**：如果修改很小且无跨模块影响，可简化或省略

## 无问题报告

如果审查未发现任何问题：

```markdown
# 📋 Code Review 报告

## 修改概要
{同上}

## 审查结果

✅ 本次修改未发现明显问题。

### 审查覆盖
- [x] 逻辑 Bug
- [x] Crash 风险
- [x] 内存泄漏
- [x] 性能问题
- [x] 线程安全
- [x] 资源管理

### 备注
{可选：对代码质量的正面评价或小建议}
```
