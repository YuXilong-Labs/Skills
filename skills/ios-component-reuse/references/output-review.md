## 严重级别规则

### 证据完整度分级（先判证据，再判严重级别）

| 级别 | 条件 |
|------|------|
| 完整 | `search_component` + `get_component_api` + (`get_class_detail` / `read_source` / `find_usage_example`) |
| 受限 | 存在 `api_only` 或上下文不足，无法完成源码/示例验证，但 API 证据充分 |
| 不足 | 只有模糊检索，无 API 证据（此时不应下高置信结论） |

### 高（阻塞）

满足任一项通常应阻塞：
- 新增基础能力重复实现且已有稳定组件（证据完整）
- 绕过统一网络/路由/图片基础设施
- 明确可替换 API 已给出，但仍坚持平行实现且无合理豁免

### 中（需整改）

- 实现方式与统一组件不一致，形成维护分叉风险
- 缺少关键错误处理/埋点一致性，替换后行为风险较大
- 证据受限但高度疑似重复实现（需补上下文确认）

### 低（建议优化）

- 命名、注释、调用风格与组件规范不一致
- 轻微冗余封装但暂未造成明显维护风险

### 豁免要求

豁免必须写清：
1. 当前组件缺失点（具体到能力）
2. 临时方案范围（模块/页面/调用点）
3. 后续收敛计划（责任人 + 时间）

### 阻塞判定注意事项

- 证据不足时不要直接判高风险阻塞，应先补证据或标记"需补充信息"
- `api_only` 不等于不能判定，但需在结论中写明证据受限
